# Bricked Up - MVP Implementation Plan

## Context
Build an iOS app replicating Brick's core screen time control features using personal NFC chips. The user wants to take back control of their screen time. The existing project is a fresh SwiftUI scaffold (Hello World template, Xcode 26.4, iOS 26.4).

## Top 6 MVP Features
1. **NFC Tap-to-Lock/Unlock** — physical chip toggles blocked/unblocked state
2. **App Blocking** — Shield selected apps using Apple's Screen Time API (FamilyControls/ManagedSettings)
3. **Website Blocking** — Block distracting websites in Safari and browsers via WebContentRestrictions
4. **Custom Blocking Modes** — Named profiles (Work, Sleep, Study) with different blocked apps AND websites per mode
5. **Scheduling** — Auto-activate modes at set times/days via DeviceActivity framework
6. **Usage Tracking & Streaks** — Dashboard showing bricked time, streaks, and session history

## Wave 0 — Foundation (sequential; blocks everything else)

Nothing else starts until this is done: capabilities, App Group, SwiftData models, and FamilyControls authorization.

### Xcode Manual Steps (must be done in Xcode UI)
- Add capabilities to main target: **Family Controls**, **NFC Tag Reading**, **App Groups** (`group.com.sam.bricked-up`)
- Add `NFCReaderUsageDescription` to Info.plist
- Create 3 extension targets (each with Family Controls + App Groups capabilities):
  - `DeviceActivityMonitorExtension`
  - `ShieldConfigurationExtension`
  - `ShieldActionExtension`

### New Files
- `Shared/AppConstants.swift` — App Group ID, shared UserDefaults helper
- `Shared/AppState.swift` — Cross-process state (locked/unlocked, active mode) via App Group UserDefaults
- `Shared/Models/NFCChip.swift` — SwiftData model (id, tagIdentifier, name, dateRegistered)
- `Shared/Models/BlockingMode.swift` — SwiftData model (id, name, iconName, serialized FamilyActivitySelection including web domains, customBlockedDomains [String], isActive)
- `Shared/Models/BrickSession.swift` — SwiftData model (id, modeId, startTime, endTime)
- `Shared/Models/ModeSchedule.swift` — SwiftData model (id, modeId, start/end time, activeDays, isEnabled)

### Modify
- `bricked_upApp.swift` — Add SwiftData container (stored in App Group directory), request FamilyControls auth

### Verification
- App builds and launches on physical device, FamilyControls authorization prompt appears

## Wave 1 — Two parallel tracks (after Wave 0)

These can run **in tandem** by two people (or alternating days). Define a small **BrickingService** contract up front: e.g. `toggleLock(for mode:)` calls into shielding and updates `AppState` + sessions so neither track blocks the other for long.

### Track 1A — NFC tap & chip registration
**Depends on:** Wave 0 only.

### New Files
- `Services/NFCService.swift` — `NFCTagReaderSession` with `.iso14443` polling to read hardware UID (secure, non-clonable)
- `Services/BrickingService.swift` — Central orchestrator: validates chip, toggles state, applies/removes shields, creates sessions (stub shield calls behind a protocol until Track 1B lands)
- `Views/Onboarding/NFCRegistrationView.swift` — Scan + register NFC chip on first use

### Key Detail
Use `NFCISO7816Tag.identifier` (hardware UID) not NDEF content — it's burned into the chip and can't be cloned.

### Verification
- Scan NFC tag, see UID logged; register chip; re-scan and app recognizes it; state toggles on tap (even if shield application is still stubbed)

---

### Track 1B — App & website blocking (Screen Time API)
**Depends on:** Wave 0 only. Does **not** require NFC to test—use a temporary debug button to apply/remove shields.

### New Files
- `Services/ShieldingService.swift` — Wraps `ManagedSettingsStore`; applies/removes shields per mode using named stores. Handles both app shields AND web content restrictions.
- `Views/Modes/AppSelectionView.swift` — Wraps `FamilyActivityPicker` for app selection
- `Views/Modes/WebsiteBlockListView.swift` — UI for managing blocked website domains per mode

### Extension Files (auto-generated by Xcode, then customized)
- `ShieldConfigurationExtension.swift` — Custom shield UI ("This app is bricked")
- `ShieldActionExtension.swift` — Primary action: "Stay Bricked" → `.close`

### How Website Blocking Works
- `ManagedSettingsStore` supports `store.shield.webDomains` to block specific web domains
- `FamilyActivitySelection` includes `webDomainTokens` — the FamilyActivityPicker lets users select web domains alongside apps
- Additionally, use `store.webContent.blockedByFilter` with `WebContentSettings.FilterPolicy` for URL-level blocking
- Blocked domains show the same shield overlay in Safari and any browser using `WKWebView`
- Users can also manually enter custom domains (e.g., "reddit.com", "twitter.com") which get added to the `WebDomain` block list

### Key Details
- `ManagedSettingsStore` persists across app launches/reboots. Shields remain even if app is killed — prevents bypass.
- Web domain blocking works in Safari automatically. Third-party browsers that use `WKWebView` (Chrome, Firefox, etc.) also respect `ManagedSettings` web content restrictions.
- `FamilyActivityPicker` already supports web domain selection — the same picker can select both apps and domains.

### Verification
- Select apps AND websites via FamilyActivityPicker; debug toggle blocks them all
- Blocked apps show shield overlay; blocked websites show restriction page in Safari
- Manually added domains (e.g., reddit.com) are also blocked
- Debug toggle unblocks everything

### Merge (Wave 1)
- Wire **BrickingService** to real **ShieldingService** (replace stubs).
- End-to-end: NFC tap blocks/unblocks selected apps and sites for the active mode.

## Wave 2 — Three parallel tracks (after Wave 1 merge)

All three assume **BlockingMode** persistence, **ShieldingService**, and **BrickingService** are integrated. Teams should agree on **UserDefaults keys** for serialized `FamilyActivitySelection` per mode (used by extensions) before splitting work.

### Track 2A — Custom blocking modes & home UI
**Depends on:** Wave 1 merge (modes must apply real shields).

### New Files
- `Views/Modes/ModeListView.swift` — List of modes (up to 10), swipe to delete
- `Views/Modes/ModeEditorView.swift` — Create/edit mode: name, icon, app selection, website selection (FamilyActivityPicker + manual domain entry)
- `Views/Components/IconPickerView.swift` — SF Symbol grid picker
- `Views/Home/HomeView.swift` — Main screen: state indicator, mode selector, NFC prompt, active session timer

### Verification
- Create multiple modes with different app/website sets; bricking with each mode blocks only that mode's apps and websites

---

### Track 2B — Scheduling
**Depends on:** Wave 1 merge; **needs** serialized mode payloads in App Group (coordinate with Track 2A on when modes are saved to UserDefaults).

### New Files
- `Services/ScheduleService.swift` — Registers/unregisters `DeviceActivitySchedule` with `DeviceActivityCenter`
- `Views/Schedule/ScheduleListView.swift` — Schedules per mode with enable/disable toggles
- `Views/Schedule/ScheduleEditorView.swift` — Time pickers, day-of-week selector, mode picker

### Extension File
- `DeviceActivityMonitorExtension.swift` — `intervalDidStart`: apply shields; `intervalDidEnd`: remove shields (reads mode data from App Group UserDefaults)

### Key Detail
Extensions have ~6MB memory limit. Keep them minimal — read only from UserDefaults, no SwiftData imports. Store serialized `FamilyActivitySelection` in App Group UserDefaults (keyed by mode ID) alongside SwiftData.

### Verification
- Create schedule; apps auto-block/unblock at scheduled times; manual NFC override still works during scheduled blocks

---

### Track 2C — Usage tracking & streaks
**Depends on:** `BrickSession` writes from **BrickingService** (Wave 1). Can implement UI with mock data until sessions are wired, then swap in real **StatsService**.

### New Files
- `Services/StatsService.swift` — Queries BrickSession data: daily totals, streaks, weekly aggregates
- `Views/Stats/StatsView.swift` — Dashboard: current streak, today's time, weekly bar chart (Swift Charts), lifetime hours
- `Views/Stats/SessionHistoryView.swift` — Session list grouped by date

### Verification
- After several brick/unbrick cycles, stats display correctly; streak resets on missed days

## Architecture Notes

| Concern | Approach |
|---------|----------|
| Cross-process data | App Group UserDefaults (not shared SwiftData — too complex for MVP) |
| NFC security | Hardware UID via `NFCTagReaderSession` + `.iso14443`, not NDEF |
| Store persistence | One `ManagedSettingsStore` per mode (named `mode-{id}`) |
| Concurrency | Project uses `MainActor` default isolation; mark services `nonisolated` where needed |
| Serialization | `FamilyActivitySelection` is `Codable` → JSON encode to Data |
| Testing | Must use physical device — NFC and FamilyControls don't work in simulator |

## Required Frameworks
CoreNFC, FamilyControls, ManagedSettings, DeviceActivity, SwiftData, Charts

## Parallel execution summary

| When | Who can work in parallel |
|------|---------------------------|
| **Wave 0** | One person only (foundation). |
| **Wave 1** | **Track 1A (NFC + BrickingService)** and **Track 1B (Shielding + extensions + pickers)** together; merge before Wave 2. |
| **Wave 2** | **Track 2A (modes + home)**, **Track 2B (scheduling + DeviceActivity extension)**, **Track 2C (stats)** together; align on UserDefaults keys for modes/schedules. |
| **Wave 3** | Usually one owner (navigation + onboarding) once core tabs are reviewable. |

**Critical path (shortest sequence if only one developer):** Wave 0 → Wave 1 (1A then 1B, or 1B before 1A if you prefer testing shields first) → merge → Wave 2 can follow 2A → 2B → 2C or interleave 2C early if session logging is stable → Wave 3.
